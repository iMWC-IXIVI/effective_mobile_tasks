# Домашнее задание: FastAPI микросервис для торговых данных
> __Необходимо реализовать микросервисное приложение на FastAPI, которое будет использовать данные из таблицы spimex_trading_results (продолжение задания из stage_4) и отдавать их в формате JSON с кэшированием через Redis.__

## База данных и модель данных
### Таблица spimex_trading_results:
> __Используйте данные, загруженные в предыдущем задании (stage_4). Таблица должна содержать информацию о торгах с биржи Spimex, включая дату, продукт, объем, цену и другие характеристики торговых операций.__

### Основные поля таблицы:
```text
id - уникальный идентификатор записи
date - дата торгов
oil_id - идентификатор нефтяного продукта
delivery_type_id - тип поставки
delivery_basis_id - базис поставки
volume - объем торгов
total - общая сумма
count - количество сделок
```

### Требования:
* __Использовать SQLAlchemy для работы с БД (async/await)__
* __Создать Pydantic модели для валидации данных__
* __Реализовать пул соединений к PostgreSQL__
* __Добавить индексы для оптимизации запросов__

## API Endpoint:
* ### get_last_trading_dates
___GET /api/v1/trading/dates/___
> __Возвращает список дат последних торговых дней с возможностью фильтрации по количеству последних дней.__

| Параметр  |  Тип  | Обязательный |                        Описание                        |
|:---------:|:-----:|:------------:|:------------------------------------------------------:|
| last_days | 	int  |     	Нет     | 	Количество последних торговых дней (по умолчанию: 30) |

#### Кэширование
> __Результаты кэшируются в Redis до 14:11, затем происходит сброс кэша.__

#### Обоснование параметров:
> __last_days__ - опциональный, так как пользователь может хотеть получить все даты или ограниченный период
> 
> Значение по умолчанию 30 дней обеспечивает баланс между актуальностью и производительностью

* ### get_dynamics
___GET /api/v1/trading/dynamics/___
> __Возвращает список торгов за заданный период с возможностью фильтрации по различным параметрам.__

|     Параметр      | Тип  | Обязательный |             Описание             |
|:-----------------:|:----:|:------------:|:--------------------------------:|
|    start_date     | date |      Да      |      Начальная дата периода      |
|     end_date      | date |      Да      |      Конечная дата периода       |
|      oil_id       | int  |     Нет      | Идентификатор нефтяного продукта |
| delivery_type_id  | int  |     Нет      |           Тип поставки           |
| delivery_basis_id | int  |     Нет      |          Базис поставки          |

#### Кэширование
> __Результаты кэшируются в Redis до 14:11 на основе комбинации всех параметров запроса.__

#### Обоснование параметров:
> __start_date__ и __end_date__ - обязательные, так как без них невозможно определить период анализа
> 
> __oil_id__, __delivery_type_id__, __delivery_basis_id__ - опциональные, позволяют гибкую фильтрацию
> 
> Без фильтров возвращаются все торговые операции за период

* ### get_trading_results
___GET /api/v1/trading/results/___
> __Возвращает список последних торгов с возможностью фильтрации по характеристикам продукта.__

|     Параметр      | Тип | Обязательный |                                Описание                                 |
|:-----------------:|:---:|:------------:|:-----------------------------------------------------------------------:|
|      oil_id       | int |     Нет      |                    Идентификатор нефтяного продукта                     |
| delivery_type_id  | int |     Нет      |                              Тип поставки                               |
| delivery_basis_id | int |     Нет      |                             Базис поставки                              |
|       limit       | int |     Нет      | Максимальное количество результатов (по умолчанию: 100, максимум: 1000) |

#### Кэширование
> __Результаты кэшируются в Redis до 14:11 на основе комбинации параметров фильтрации.__

#### Обоснование параметров:
> Все параметры фильтрации опциональные - позволяет получить как конкретные данные, так и общую картину
> 
> __limit__ - с дефолтным значением 100 предотвращает перегрузку системы
> 
> Максимум 1000 результатов обеспечивает баланс между удобством и производительностью

## Кэширование с Redis
### Требования к кэшированию
* Использовать Redis для кэширования всех трех endpoints
* Ключ кэша должен генерироваться на основе всех параметров запроса
* Время жизни кэша - до 14:11 каждого дня
* В 14:11 происходит полный сброс (flush) всех кэшированных данных
* Реализовать автоматическую очистку кэша по расписанию
* Логировать попадания в кэш (cache hits) и промахи (cache misses)

### Логика кэширования:
* При поступлении запроса проверять кэш по ключу
* Если данные найдены в кэше - возвращать их (cache hit)
* Если данных нет - выполнять запрос к БД и сохранять результат в кэш
* Использовать JSON-сериализацию для хранения данных в Redis
* Реализовать инвалидацию кэша в 14:11 через background task

### Формат ключа кэша:
* trading:dates:last_days:{value}
* trading:dynamics:{hash_of_params}
* trading:results:{hash_of_params}
